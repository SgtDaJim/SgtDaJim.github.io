运维之路 - 运维原则与奇技淫巧[持续更新]

Posted by Ro$es in Technology

Tags Linux  Ops 

现总计35条

  • 做改动前做备份，无论多小的修改，都要做备份,因为备份是救命用的。

  • 做改动之前，先小范围测试，百分之一万确定没有任何问题时，再全范围修改。（此处可以参考版本发布流程规范）

  • 在执行命令或者其他操作时，前后三思，做好最坏打算，同时做好回滚准备。

  • 在执行别人给的命令和操作时，先粘贴到别的地方看清楚，有没有错误，确定没有错误再执行。

  • 在查问题时，首先要想到查看日志。（没有日志？那就提前创造日志，别等出了事才知道慌。）

  • 处理技术问题的基本思路:

    1. 怎么发现问题的
    2. 怎么处理的
    3. 影响多大
    4. 怎么预防/避免

  • 做出改动前，先认真考虑影响。直接改动的影响是什么？（业务影响、维护影响等）影响面广、影响重要应用的，请求多人确认；影响面小的，直接干。不能确定影响的，找人当面询问。（最低限度发邮件）

  • 凡事有交代，件件有着落，事事有回音。（做运维如此，做人更应如此。）

  • 善用md5sum，可以快速找出哪些文件是不一样的。（特别是用Ansible对比多台机器上的配置文件时，非常方便直观）

  • 查看mongodb运行状态 mongostat

  • ss(Socket Statistics的缩写)命令可以用来获取 socket统计信息，此命令输出的结果类似于 netstat输出的内容，但它能显示更多更详细的 TCP连接状态的信息，且比 netstat 更快速高效。
    ss -s 输出统计信息。

  • 在zabbix server上运行zabbix_get，可以执行一个key。这样可以检查key是否设置正常。
    zabbix_get -s host/IP [-p 端口] -k key

  • 善用screen，防止断网或者其他突发情况时，正在ssh上做的东西被中断导致严重后果！！ screen -R 名称

  • 安装PHP扩展时，可以选择使用PHP自带PECL扩展库安装。
    PECL
    pecl命令

  • nginx -t 检测配置文件语法错误，同时会显示主配置文件路径

  • nginx -s reload 工作过程：
    master进程检查配置文件是否正确。若错误，则返回错误信息,使用原来的配置继续工作；若正确，则启动新的worker进程,这些worker使用新的配置文件，新的请求也会分发到新的worker;旧的worker在处理完已分配的旧的请求后，就会被关闭。在这个过程，nginx的工作完全不受影响。

  • python 连接数据库时，为了能重复使用cursor，避免在多次连接数据库上花费大量时间，可以将数据库的连接做成class。

  • curl -I 可以输出http header，-s 静默模式，不输出进度表

  • 当机器无端端ssh连不上，但机器存活时，可以检查cpu和内存情况，然后再考虑ssh服务情况。

  • zabbix中端口监控，type应该选active，使监控更加准确。

  • 在修改crontab时，尽量跳到该用户下，使用crontab -e命令进行修改。

  • 在shell中，冒号“:”可以当占位符使用。（与Python的“pass”相同）

  • Zabbix的日志监控
    Zabbix会跟踪日志的大小、最后的修改时间，有时会有inode号或者indexes号。Zabbix将这些数据保存在数据库中。当Zabbix分析文件时，会检查这些数据，判断日志进行过最新修改后已经被分析过了。如果（最新修改时间已经改变，而且根据日志的大小判断上一次分析到的地方，从那个地方开始）有新的部分没有分析，就会接着分析，满足条件时触发trigger。

  • logtail 分析日志，并记录下分析到日志的哪个地方。

  • sh -x 脚本名 可以在运行脚本时实现shell脚本逐条语句的跟踪。

  • zgrep 可以不解压日志直接抓关键字

  • 统计每分钟日志数 awk '{a[$2]++}END{for(i in a){print i" "a[i]}}'

  • centos中，nginx做转发时，记得修改SELinux配置。 setsebool -P httpd_can_network_connect 1

  • 当遇到4xx问题时，先考虑文件是否存在，然后考虑nginx配置是否正确。遇到5xx问题时，先考虑后端不返回。

  • 在ssh时，提示permission denied(publickey)，其中一个点是可以尝试留意下.ssh/config中的内容。有时候这里会出问题。

  • tar命令中， -C 参数意思是change to directory，可以做到跳到某目录然后打包该目录下所有文件的效果。
    tar -C ~/test/111/ -zcvf 123.tar.gz ./

  • logrotate中，create是新建一个日志文件，copytruncate是复制文件，再把原文件清空
    所以对nginx日志使用create时，要nginx -s reload，让nginx重新识别日志文件，防止日志无法写入

  • nginx在处理请求时，当有多个配置中，使用相同域名和相同端口，则访问域名时，nginx将请求发给default server（在listen字段加default）。当没设定default server时，若配置都在同一个配置文件中，请求则发给第一个配置；若在不同配置文件中，则按ascii码，找最前的配置文件。当然，listen字段中，同ip同端口只能有一个设定为default server。

  • php-fpm的process_control_timeout参数应该设置为适当的值。php-fpm在reload时，process_control_timeout过小，会导致请求处理时间不够，出现502错误。当process_control_timeout的值过大时，空闲的fastcgi进程关闭，正在处理请求的进程则继续处理，若处理速度过慢，则会导致fastcgi进程不够而导致请求大量堆积。（所以迫切需要重启php-fpm的，还是把机器下流量，然后php-fpm restart吧。）

  • 使用CDN时，当大量CDN节点进行回源，将会对源服务器造成巨大的压力。此时，可以选择与CDN运营商沟通（如果是自建CDN就自己处理），只让少量（2、3台都可以了）的CDN节点进行回源，然后其他CDN节点到那些已经回源的节点上进行更新。这样源站的压力就会大大减轻。
